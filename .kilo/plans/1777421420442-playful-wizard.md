# Full-App Issues and Errors Audit Plan

## Goal
Find and prioritize issues across the Watum app without making application changes during the audit pass. The deliverable is a ranked findings report with file/line references, impact, severity, reproduction or verification steps, and recommended fixes.

## Scope
Audit the full application:

- Frontend UI and Svelte state management in `src/routes/+page.svelte`, `+layout.svelte`, app components, and UI primitives.
- SvelteKit remote functions in `src/routes/**/data.remote.ts`.
- Server utilities in `src/lib/server/**`, including auth, database pooling, conflict audit, search, and SQL helpers.
- Validation schemas in `src/lib/validations/**`.
- Test setup, Playwright coverage, demo/test routes, scripts, and CI-readiness.
- Accessibility, responsive behavior, theming, UI performance, data integrity, security, authorization, reliability, and maintainability.

## Non-Goals

- Do not fix issues during the audit unless explicitly requested after the report.
- Do not add dependencies or rework architecture during the discovery pass.
- Do not claim a mathematically complete list of every possible bug. The report will cover issues discoverable through static analysis, existing checks, targeted runtime tests, and manual browser inspection.

## Known Context

- Stack: SvelteKit 2.x, Svelte 5, Bun, MariaDB, TypeScript, Playwright.
- Current design context exists in `.impeccable.md`: academic operations product, academic/sleek/calming, clarity over decoration, Indonesian UI.
- `bun run check` recently passed with `0 errors and 0 warnings` after accessibility updates, but deeper semantic/runtime issues may still exist.
- The main app route is very large (`src/routes/+page.svelte`, ~12.5k lines), which increases coupling and regression risk.

## Preliminary Issues To Verify
These are early read-only findings from code exploration and should be validated before final reporting.

### P0/P1 Candidate Findings

- Enrollment conflict checks may be incorrectly scoped by only day/time/resource and not `academic_year`/`semester`, causing false rejections across terms.
- Duplicate enrollment checks use `student_id + course_id + semester` but omit `academic_year`, potentially blocking valid repeats in a later year.
- `createEnrollment` allows `STUDENT` users to create schedules with arbitrary room/day/time, which may be a data-integrity and authorization issue if students should only enroll in existing offerings.
- Conflict and duplicate checks happen before transactions and without visible uniqueness constraints/locks, allowing race conditions under concurrent submissions.
- Grade creation appears to have a check-then-insert uniqueness race unless the database has a unique constraint on `grades.enrollment_id`.
- Bulk user role updates may assign `STUDENT`/`LECTURER` roles without enforcing valid linked `student_id`/`lecturer_id`.
- Test coverage is almost absent for production flows; existing Playwright test only checks a demo page.
- Hardcoded DB credentials may exist in ad-hoc performance/audit scripts and should be verified as a security issue.

### P2/P3 Candidate Findings

- Entity links with `role="button"` inside row buttons resolve `svelte-check` warnings but still create nested-interactive semantics; consider restructuring rows in a future pass.
- Custom comboboxes lack full ARIA combobox semantics, active-descendant management, arrow-key navigation, and Escape handling.
- Multiple queued timers in `+page.svelte` need teardown verification to prevent state updates after navigation.
- Global `window.fetch` monkey-patching in auth/loading interceptors is fragile under HMR/import-order changes.
- Calendar event content uses raw HTML with manual escaping, which is safe only while every interpolated value is consistently escaped.
- Mobile calendar uses a very wide minimum width and should be manually validated for usability.
- Bulk endpoints and unpaginated helper queries need bounds review.
- MariaDB boolean fulltext search input should be checked for operator/special-character behavior.

## Audit Method

### 1. Baseline State and Existing Changes

- Inspect `git status` and current diffs to distinguish existing user/agent changes from app defects.
- Review package scripts, config, Playwright config, Vite/Svelte config, and deployment scripts.
- Confirm whether generated files or local-only artifacts should be excluded from findings.

### 2. Static Checks

Run and capture outputs:

- `bun run check`
- `bun run lint`
- `bun run build`
- `bun run test:e2e`

If any command mutates generated files, note it and separate generated noise from source findings.

### 3. Frontend/UI Audit

Inspect and manually test:

- App shell navigation for Admin/Lecturer/Student roles.
- Enrollment builder, editor search comboboxes, schedule filters, bulk actions, pagination, and detail panels.
- Keyboard-only navigation through sidebar, entity links, list rows, comboboxes, builder steps, forms, calendar, and pagination.
- Screen reader semantics for loading spinner, selected rows, comboboxes, conflict states, validation errors, and entity links.
- Mobile breakpoints: `320x568`, `390x844`, `430x932`, tablet, desktop.
- Console errors/warnings during login, view changes, filtering, editing, and logout.

Report categories:

- Accessibility
- Responsive/mobile
- State management
- Form binding/validation display
- Runtime errors
- UI performance and excessive rerenders
- Theming/contrast
- AI/design anti-patterns only where they affect quality or consistency

### 4. Server/Auth/Security Audit

Inspect and test critical paths:

- Auth lifecycle: login, refresh rotation, logout, missing/expired refresh, invalid access token retry.
- Role guards for every remote function and command.
- CSRF/origin behavior behind proxy settings.
- JWT config defaults and production hardening requirements.
- Refresh-token storage, revocation, and concurrency behavior.
- Hardcoded credentials/secrets in scripts, docs, tests, and examples.
- XSS blast-radius risks from in-memory access tokens and raw HTML calendar rendering.

### 5. Data Integrity and Business Rule Audit

Verify policy and implementation for:

- Enrollment creation/update rules by role.
- Whether schedules represent class offerings or individual student enrollments.
- Conflict checks across room/student/lecturer dimensions.
- Academic-year and semester scoping for conflicts and duplicates.
- Same-course multi-student behavior.
- Grade uniqueness and grade lock behavior.
- Bulk operations for users, courses, students, lecturers, rooms, enrollments, and grades.
- Trigger-dependent denormalized audit fields and drift recovery.

Use DB/migration inspection first; if execution is approved and a safe database is available, add targeted integration tests or manual scenarios.

### 6. Performance and Reliability Audit

Inspect and benchmark where possible:

- Large list query paths and cursor behavior.
- Conflict audit cache keys, invalidation, TTL behavior, and in-flight deduplication.
- Database pool limits, unbounded queues, transaction sizes, and retry behavior.
- Bulk endpoint transaction duration and item limits.
- Main Svelte component reactive graph for avoidable refresh loops and timer leaks.
- Bundle/build output for obvious heavy imports or dead code.

### 7. Test Coverage Audit

Report current coverage gaps and propose a test suite:

- Unit tests for validation, grade calculation, time helpers, search helpers, and conflict helper logic.
- Integration tests for remote functions, role matrix, enrollment conflicts, grade creation, and auth lifecycle.
- E2E tests for login, navigation, CRUD happy paths, enrollment editing, filtering, conflict errors, bulk actions, and mobile nav.
- Test hygiene for `/test` and `/demo` routes in production.

## Severity Rubric

- P0 Blocking: breaks core task completion, creates data corruption, or exposes critical security risk.
- P1 Major: significant business-rule bug, authorization flaw, WCAG AA failure, or high-probability production failure.
- P2 Minor: incorrect edge case, poor usability, maintainability risk, performance risk with workaround.
- P3 Polish: cleanup, consistency, small UX improvement, or future hardening.

## Final Report Structure

1. Executive summary with total P0/P1/P2/P3 counts.
2. Top critical findings to address first.
3. Findings table: severity, area, file/line, issue, impact, verification, recommendation.
4. Detailed findings grouped by category.
5. Test coverage gaps and recommended tests.
6. Positive findings to preserve.
7. Suggested remediation order.

## Execution Notes

- Keep audit findings factual and reproducible; avoid speculative claims without marking them as “needs verification”.
- Prefer minimal recommended fixes, especially around large shared code paths.
- Do not refactor the app during the audit. Remediation should happen in separate, focused implementation passes.
- If a safe seeded database is not available, mark DB-dependent findings as static-analysis candidates and provide exact scenarios to verify later.

## Definition of Done

- All configured checks have been run or explicitly skipped with reason.
- Critical user flows have been manually or programmatically exercised.
- Findings are ranked by severity and include file/line references where possible.
- Every P0/P1 finding has a concrete reproduction or verification path.
- The user receives a clear next-action list for fixing the highest-impact issues first.
